# Update External Skills

Runtime-manage the `npx skills add`-installed skills in this project. Not a skill-creation tool — a skill **refresher** with reporting.

## When this skill runs

This project has a `skills-lock.json` at its root, generated by `npx skills`. Each entry looks like:

```json
{
  "skills": {
    "claude-api": {
      "source": "anthropics/skills",
      "sourceType": "github",
      "computedHash": "ae0442705d2ce9338f91fffafa423172abfa85c7c9b85f13dece6cdc9b1681cc"
    }
  }
}
```

The `source` points to a GitHub repo; `computedHash` is the content hash of the installed skill. `npx skills update` refreshes these from upstream. This skill wraps the update with:
- GitHub metadata enrichment (stars, pushed_at)
- Interactive per-skill selection
- Before/after hash diff → real drift report
- Auto git commit+push when drift is real

## How to invoke

From project root:

```bash
python3 .claude/skills/update-external-skills/scripts/update.py
```

Or via slash command: `/update-external-skills`.

The script walks up from CWD looking for `skills-lock.json`, so it can be invoked from any subdirectory of the project.

## What the script does

1. **Locate** `skills-lock.json` (fail cleanly if absent).
2. **Parse** tracked skills into an internal record set.
3. **Enrich** each record with GitHub metadata (dedup: one API call per unique repo).
4. **Present** an indexed selection table:

   ```
     #  Skill                  Repo                         Stars  Remote pushed   Status
     1  ai-seo                 searchfit/seo-skills         312    2d ago          stale
     2  b2b-saas-marketing     searchfit/saas-skills        145    5d ago          current
     3  ...
   ```
   *stale* = repo's `pushed_at` is newer than the local skill file's mtime.

5. **Prompt** the user:
   - `0` → all skills
   - `1,3,5` → specific indexes
   - `1-5` → range
   - empty → cancel, exit

6. **Execute** `npx skills update <selected> -p -y` on the chosen set.
7. **Re-read** the lock file, diff old vs new `computedHash` per skill.
8. **Report** final result: updated / unchanged / skipped, with stats.
9. **Auto-commit+push** if at least one hash actually changed. Staging is explicit (`skills-lock.json`, `.agents/skills`, `.claude/skills`) — never `git add -A`, so unrelated uncommitted work is not swept in.

## Rate limits

GitHub API without token: 60 requests/hour. Because the script deduplicates by repo, projects with many skills from few repos (e.g. `anthropics/skills`) stay well under limit. If you hit a rate-limit warning, export `GITHUB_TOKEN`:

```bash
export GITHUB_TOKEN="ghp_..."      # 5000/hr
```

No token → the script falls back gracefully; metadata columns show `no-meta` for rate-limited entries, but the update itself still runs.

## What the script does NOT do

- Install new skills — use `/find-skills` then `npx skills add <owner/repo@skill>`.
- Remove skills — use `npx skills remove <name>`.
- Update kit-shipped skills (agents/, kit skills, slash commands) — use `deploy.sh --update-skills` (or its equivalent) at the orchestration-kit level.
- Touch `.claude/settings.local.json`, `.claude/orchestration-config.json`, `.beads/`, `docs/orchestration/`, or `CLAUDE.md`.

## Presentation of script output

The script prints a fixed-width table with columns: `#`, `Skill`, `Repo`, `Stars`, `Remote pushed`, `Status`. **Show this table VERBATIM to the user** — don't reformat it into a markdown table, don't drop the `Stars` or `Status` columns. Those columns ARE the statistics the user is asking for when they invoke this skill. Compacting the table defeats the purpose of running it.

If the table appears wide in the terminal/UI, let it wrap naturally. Do not squeeze it into a narrower layout at the cost of dropping columns.

## Failure modes

| Symptom | Cause | Fix |
|---|---|---|
| `No skills-lock.json found` | Project has never run `npx skills add` | Install a skill first: `npx skills add <owner/repo@skill> -p -y` |
| All status show `no-meta` | GitHub rate limit hit | Export `GITHUB_TOKEN` or wait 1 hour |
| `npx skills update` exits non-zero | Network / registry / version issue | Re-run `npx skills update <name>` directly to see full error |
| Script prompts but terminal is non-interactive (CI) | No TTY | Currently not supported; extend with `--all` flag if needed for CI |
| Git push fails | No upstream / auth | Commit is preserved locally; push manually later |

## Design notes

- **Autonomy:** the script lives inside the skill directory, so after `deploy.sh --update-skills` each project has a fresh copy. No need to clone orchestration-kit to run updates.
- **Selection UX:** mirrors `git add -i` — indexed table, number/range input. Familiar pattern.
- **Dedup on GitHub API:** 43 skills across 8 projects typically ≈ 10–15 unique repos. One API call each. Safe for anon 60/hr.
- **Hash-based drift:** `computedHash` in lock is the ground truth. Pre/post comparison = reality, independent of `npx skills` exit code nuances.
- **Explicit git paths:** lesson from deploy.sh — `git add -A` would sweep unrelated user work into the auto-commit. Staging only `skills-lock.json` + skill directories keeps the commit focused.
